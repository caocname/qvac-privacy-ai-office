# 数据存储位置与安全保护机制

## 一、存储位置总览

项目所有运行时数据统一存储在 `data/` 目录下（开发模式为项目根目录 `data/`，生产模式为 `%APPDATA%/qvac-assistant/data/`）。

| 路径 | 内容 | 加密状态 |
|---|---|---|
| `data/qvac.db` | SQLite 主库（知识库元数据、会话、聊天记录、RAG 分片、ASR 归档） | 敏感字段 AES-256-GCM 列级加密 |
| `data/audit_logs.db` | 审计日志（独立 SQLite） | 明文（JSON 结构化日志，用于赛事审计评估） |
| `data/uploads/` | 用户上传的原始文档（PDF/DOCX/TXT/MD） | 文件名映射加密，文件正文明文（用于 FAISS 索引） |
| `data/faiss_indices/` | FAISS 向量索引文件 | 明文（赛事要求允许不加密） |
| `data/models/` | 本地模型文件（.gguf / .bin） | 不加密（静态模型，非用户数据） |

### 详细说明

**`data/uploads/`** — 用户通过知识库界面上传的所有文档原始文件。文件以 UUID 重命名存储，原始文件名经 AES-256-GCM 加密后存储在 `knowledge_base.file_path` 字段中。

**`data/faiss_indices/`** — 每个知识库文档经 512 Token 分片、Embedding 向量化后，生成的 FAISS 向量索引文件。按 `file_id` 组织。

**`data/qvac.db`** — 包含 6 张核心表：
- `knowledge_base` — 文档元数据，敏感字段 `file_path` 加密
- `knowledge_folders` — 文件夹结构
- `asr_archive` — ASR 转写归档，敏感字段 `audio_path`、`transcribed_text` 加密
- `sessions` — 聊天会话
- `chat_records` — 聊天消息
- `rag_chunks` — RAG 分片内容

**`data/audit_logs.db`** — 记录所有系统操作（推理、检索、安全事件、文件操作），每条含 `absolute_datetime` 和 `relative_timestamp_ms`。

## 二、安全保护机制

### 2.1 主密钥管理（三层架构）

```
Windows 凭据管理器 (User 级普通凭据)
  └── kvac-text / db-master-key → SHA-256 派生
       └── 32-byte AES-256-GCM Key → 加密 SQLite 敏感字段
```

1. **首次启动**：自动生成 32-byte 随机主密钥，注入 Windows 凭据管理器（`Credential Manager` → `Windows 凭据` → `通用凭据`）
2. **正常运行**：从凭据管理器读取 → SHA-256 派生 → 用于 AES-256-GCM 加解密
3. **密钥丢失**：系统自动检测凭据丢失（30s 轮询），弹出灾备恢复弹窗

### 2.2 灾备恢复方案

| 恢复方式 | 说明 |
|---|---|
| `.key` 文件 | 64 位 hex 编码的恢复密钥，首次启动引导导出 |
| 12 位中文助记词 | 备选恢复方案，可从助记词还原主密钥 |

两种恢复方式均通过 `系统设置 → 导入恢复密钥` 或启动时自动弹出的灾备弹窗完成。

### 2.3 列级加密（AES-256-GCM）

数据库敏感字段使用 **AES-256-GCM**（认证加密）进行列级加密：

- 每次加密使用随机 12-byte nonce
- 密文格式：`nonce(12B) + ciphertext + auth_tag(16B)` → hex 编码存入 SQLite
- 加密字段：`knowledge_base.file_path`、`asr_archive.audio_path`、`asr_archive.transcribed_text`
- 非加密字段：文件名、文件大小、会话标题、聊天内容（文本）、分片内容

### 2.4 物理安全

- 所有数据存储在用户设备本地，**零外网出站连接**
- IPC 通信仅绑定 `127.0.0.1`，不向局域网或广域网开放端口
- Kill Switch 网络探针每 5s 检测一次，发现异常立即锁定（状态码 999）

## 三、数据生命周期

### 3.1 文档上传流程

```
用户拖拽/选择文件
  → 前端校验格式（PDF/DOCX/TXT/MD）
  → POST /api/v1/knowledge/upload（状态码: 110 → 120）
  → 文件以 UUID 重命名保存至 data/uploads/
  → QVAC SDK 结构化提取文本
  → 512 Token 分片（chunk_overlap=128）
  → Embedding 向量化（gte-large_fp16.gguf）
  → FAISS 索引写入 data/faiss_indices/
  → 加密元数据写入 data/qvac.db knowledge_base 表
  → 状态码: 100 IDLE
```

### 3.2 文档删除流程（安全擦除）

```
用户点击删除
  → DB is_deleted = 1（原子打标，立即生效，UI 不可见）
  → 后台 ThreadPoolExecutor 异步执行 os.remove（不阻塞主线程）
  → 启动时 startup_guard.py 自检：
       DELETE FROM knowledge_base WHERE is_deleted=1
  → 关联 FAISS 索引同步清理
```

### 3.3 全量清理（新增功能）

系统设置 → 数据清理 → 「清理所有数据」按钮（3 秒倒计时防呆确认）→ `POST /api/v1/system/clear-all`：

1. 清空全部数据库表（保留表结构）
2. 清空审计日志
3. 异步删除 `data/uploads/` 下所有文件
4. 异步清空 `data/faiss_indices/` 目录
5. 写入审计日志记录清理操作

## 四、隔离模式

知识库文档支持三种隔离级别：

| 模式 | `isolate_mode` | 说明 |
|---|---|---|
| 全局文档 | `global` | 所有会话可见，长期保留 |
| 会话文档 | `session` | 仅绑定会话可见 |
| 临时文档 | `temp` | 切换会话时立即销毁（显存 + 内存 + FAISS 索引） |

---

## 五、安全注意事项

1. `.key` 文件和 12 位助记词是恢复加密数据的唯一凭证，请妥善保管
2. 主密钥存储在 Windows 凭据管理器中，重装系统前务必导出 `.key` 文件
3. 物理删除操作通过 `is_deleted` 原子打标后异步执行，确保数据不可恢复
4. FAISS 向量索引不加密（赛事评审需要可评估性），但原始文件路径已加密
5. "清理所有数据"操作不可撤销，请确认已备份重要文件后再执行
